# Vendor Implementation Ecosystem

Implementations come in two forms in SimVault: **implementation links** (casual, external references) and **model definitions** (structured, collaborative, git-backed artifacts). This document covers implementation links; see [028_model_definitions_and_translators.md](028_model_definitions_and_translators.md) for the premium, translator-compatible form.

Both coexist. A casual reference (a blog post, a video, a paper) gets an implementation link. A serious, collaborative, runnable asset gets a model definition.

## How Implementation Links Work

- Each model entry has an "Implementations" section — a list of links to that model built in specific simulation software.
- An implementation link is not hosted content — it points outward to wherever that implementation lives (vendor website, GitHub repo, YouTube tutorial, university course page, etc.).
- Each link includes: software name, URL, brief description of what's covered, contributor who submitted it.

## Vendor Representation

- The registry maintains a list of known DES software platforms (AnyLogic, Simio, Arena, FlexSim, Quodsi, SIMUL8, etc.).
- Vendors are presented neutrally — no ranking, no endorsement, alphabetical or random ordering.
- Over time, the registry naturally shows which software has the most community-contributed implementations, but this is organic rather than promoted.

## Who Submits Implementation Links

- Any logged-in user — could be a vendor, a professor, a practitioner, a student.
- Vendors are welcome to submit links to their own content, same as anyone else.
- Attribution goes to the submitter, not the vendor.

## Quodsi's Role

- Quodsi appears in the vendor list alongside everyone else.
- Quodsi team can submit implementation links for models they've built, just like any other contributor.
- No preferential placement or promotion for Quodsi implementations.
